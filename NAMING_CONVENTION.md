# Правила названия файлов

## Назначение файла

Этот документ задает единые правила названий, чтобы файлы Clarus Studio было легко искать, сортировать и переносить в архив.

## Когда использовать

- При создании нового КП.
- При оформлении кейса.
- При создании заметки по клиенту.
- При сохранении проектных файлов.
- При переносе материалов в архив.
- При подготовке публичных материалов.

## Как использовать

Перед созданием файла выбрать тип документа, поставить дату, короткое название клиента или темы и понятный тип файла. Название должно объяснять, что это за файл, без открытия документа.

## 1. Зачем нужны правила названий

Единые названия помогают:

- быстро находить нужный документ;
- видеть дату создания;
- не путать версии;
- отделять активные материалы от архивных;
- сохранять порядок в проектах и КП;
- передавать работу другому человеку без дополнительных объяснений.

## 2. Общий формат

Основной принцип:

`YYYY-MM-DD_client-or-topic_file-type`

Правила:

- дата всегда в начале, если файл связан с клиентом, проектом или решением;
- слова разделяются дефисом или нижним подчеркиванием;
- технические файлы создаются без пробелов;
- публичные файлы лучше называть английским транслитом;
- внутренние документы могут иметь русское название, если это удобно команде;
- название должно быть коротким и рабочим.

## 3. Документы

Формат:

`YYYY-MM-DD_short-topic_type.md`

Примеры:

- `2026-01-10_catalog-structure_notes.md`
- `2026-01-15_lead-statuses_update.md`
- `2026-01-18_project-process_checklist.md`

## 4. КП

Формат:

`YYYY-MM-DD_client_commercial-offer.docx`

Примеры:

- `2026-01-12_business-placeholder_commercial-offer.docx`
- `2026-01-12_business-placeholder_commercial-offer_v2.docx`
- `2026-01-14_beauty-studio_commercial-offer_final.docx`

## 5. Кейсы

Формат:

`YYYY-MM-DD_client_case-study.md`

Примеры:

- `2026-01-20_business-placeholder_case-study.md`
- `2026-01-22_training-center_case-study.md`

## 6. Клиентские заметки

Формат:

`YYYY-MM-DD_client_notes.md`

В заметках фиксируются разговоры, решения, ограничения, ожидания клиента и следующие шаги.

## 7. Проектные файлы

Формат:

`YYYY-MM-DD_client_project-brief.md`

Дополнительные варианты:

- `YYYY-MM-DD_client_structure.md`
- `YYYY-MM-DD_client_materials-list.md`
- `YYYY-MM-DD_client_launch-checklist.md`

## 8. Изображения

Формат:

`YYYY-MM-DD_client_image-purpose.ext`

Примеры:

- `2026-01-16_business-placeholder_homepage-preview.png`
- `2026-01-16_business-placeholder_catalog-card.png`
- `2026-01-17_training-center_before-screen.png`

Изображение должно быть названо по назначению, а не случайным номером.

## 9. Архивы

Формат:

`YYYY-MM-DD_client-or-project_status`

Примеры:

- `2026-01-25_business-placeholder_completed`
- `2026-01-26_beauty-studio_refused`
- `2026-02-01_catalog-offer_old-version`

## 10. Черновики

Для черновиков использовать пометку `draft`.

Примеры:

- `2026-01-12_business-placeholder_commercial-offer_draft.docx`
- `2026-01-13_catalog-structure_draft.md`

Черновик не должен оставаться активным без статуса. Если документ уже отправлен клиенту, он не считается черновиком.

## 11. Запрещённые названия

Не использовать:

- `new file.md`
- `final_final.docx`
- `copy.docx`
- `client.docx`
- `text.md`
- `offer.docx`
- `123.png`
- `разное.md`
- `важное.docx`
- `последнее точно.docx`

Такие названия не объясняют дату, клиента, тип и статус файла.

## 12. Примеры правильных названий

| Тип | Правильное название |
| --- | --- |
| КП | `2026-01-12_furniture-store_commercial-offer.docx` |
| Кейс | `2026-01-20_beauty-studio_case-study.md` |
| Заметки | `2026-01-21_training-center_notes.md` |
| Структура | `2026-01-22_furniture-store_structure.md` |
| Архив | `2026-01-30_training-center_completed` |

## 13. Примеры неправильных названий

| Неправильно | Почему плохо |
| --- | --- |
| `КП новое.docx` | Нет даты и клиента. |
| `final2.docx` | Непонятно, к чему относится файл. |
| `сайт.md` | Слишком общее название. |
| `client notes.md` | Есть пробел и нет даты. |
| `photo.png` | Непонятно назначение изображения. |
